# Data Model: Strength Progression Tracker

## Entity: User
- Fields:
  - `id` (UUID, primary key)
  - `email` (string, unique)
  - `displayName` (string, optional)
  - `createdAt` (timestamp)
- Relationships:
  - 1-to-many with `WorkoutSession`
- Validation rules:
  - `email` must be valid email format

## Entity: WorkoutSession
- Fields:
  - `id` (UUID, primary key)
  - `userId` (UUID, foreign key -> User.id)
  - `startedAt` (timestamp)
  - `endedAt` (timestamp, nullable)
  - `notes` (string, optional, max 1000 chars)
  - `createdAt` (timestamp)
  - `updatedAt` (timestamp)
- Relationships:
  - many-to-1 with `User`
  - 1-to-many with `ExerciseEntry`
- Validation rules:
  - `endedAt` must be greater than or equal to `startedAt` when present

## Entity: ExerciseEntry
- Fields:
  - `id` (UUID, primary key)
  - `sessionId` (UUID, foreign key -> WorkoutSession.id)
  - `userId` (UUID, foreign key -> User.id)
  - `exerciseName` (string, max 120 chars)
  - `normalizedExerciseName` (string, max 120 chars)
  - `sets` (int)
  - `reps` (int)
  - `weight` (decimal(8,2))
  - `weightUnit` (enum: `kg` | `lb`)
  - `performedAt` (timestamp)
  - `createdAt` (timestamp)
  - `updatedAt` (timestamp)
- Relationships:
  - many-to-1 with `WorkoutSession`
  - many-to-1 with `User`
- Validation rules:
  - `exerciseName` required, trimmed, length 1..120
  - `normalizedExerciseName` required and generated by canonicalization (lowercase, collapse spaces)
  - `sets` integer range 1..100
  - `reps` integer range 1..500
  - `weight` decimal range 0.0..5000.0
  - `performedAt` cannot be in the far future beyond configured tolerance (e.g., >24h)

## Entity: ProgressComparison (derived/read model)
- Fields:
  - `exerciseName` (string)
  - `currentEntryId` (UUID)
  - `currentVolume` (decimal)  // sets * reps * weight
  - `previousEntryId` (UUID, nullable)
  - `previousVolume` (decimal, nullable)
  - `bestEntryId` (UUID, nullable)
  - `bestVolume` (decimal, nullable)
  - `deltaFromPrevious` (decimal, nullable)
  - `deltaFromBest` (decimal, nullable)
  - `statusVsPrevious` (enum: `improved` | `unchanged` | `declined` | `no-baseline`)
  - `statusVsBest` (enum: `improved` | `unchanged` | `declined` | `no-baseline`)
- Relationships:
  - Derived from `ExerciseEntry` history grouped by `userId + normalizedExerciseName`
- Validation rules:
  - Calculated server-side only; no direct persistence required

## Indexing Strategy
- `IX_ExerciseEntry_User_Normalized_PerformedAt` on (`userId`, `normalizedExerciseName`, `performedAt` DESC)
- `IX_ExerciseEntry_Session_PerformedAt` on (`sessionId`, `performedAt` ASC)
- `IX_WorkoutSession_User_StartedAt` on (`userId`, `startedAt` DESC)

## State Transitions

### WorkoutSession
- `active` -> `completed` when `endedAt` is set
- `active` -> `active` while entries are being appended/edited

### ExerciseEntry
- `draft` -> `saved` on successful validation + persistence
- `saved` -> `updated` when edited
- `saved|updated` -> `deleted` when removed by user

### ProgressComparison
- `no-baseline` -> `improved|unchanged|declined` once at least one comparable prior entry exists
